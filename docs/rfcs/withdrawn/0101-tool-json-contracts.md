# RFC 0101: Tool JSON Contract for Task Operations

## Summary

Define a strict JSON response contract for task-related tools (e.g. add/list/complete/remove/update) so clients can reliably verify outcomes, detect errors, and perform post-condition checks without scraping human output.

## Motivation

Recent usage shows tool calls returning empty output or unclear state updates, leading to fallback CLI use and uncertainty about task creation/completion. A formal response schema with explicit status, identifiers, and plan revision metadata will enable reliable automation and consistent UI/UX.

## Detailed Design

### Terminology

- **Task ID**: A canonical ULID generated by the system (no caller-provided ID).
- **Plan Revision**: A deterministic value representing the `implementation-plan.toml` content after mutation.
- **Operation Status**: One of `ok`, `warning`, or `error`.

### User Experience (UX)

- Tool calls return structured JSON that is safe to display directly or validate programmatically.
- Clients can check `status` and `planRevision` to confirm mutations.
- Clients can optionally follow with `task.list` to confirm state when `status != ok`.

### Architecture

- Task tools emit JSON conforming to the schemas below.
- The tool layer computes and returns a plan revision (hash of canonicalized TOML).
- The tool layer reports all errors in a single structured payload.

### Implementation Details

#### Common Envelope

All task tool responses MUST use this envelope:

```json
{
  "status": "ok | warning | error",
  "operation": "task.add | task.list | task.complete | task.remove | task.update | task.reorder",
  "planPath": "docs/agent-context/current/implementation-plan.toml",
  "planRevision": "<sha256 or ULID>",
  "messages": [{ "level": "info | warning | error", "text": "..." }],
  "errors": [
    { "code": "string", "message": "string", "details": { "...": "..." } }
  ]
}
```

#### task.add Response

```json
{
  "status": "ok",
  "operation": "task.add",
  "planPath": "docs/agent-context/current/implementation-plan.toml",
  "planRevision": "01K...",
  "task": {
    "id": "01KX2JH7R1H4WJ6YJ2G8V0J3K9",
    "label": "Implement tests for Reactive Collections (RFC 0100)",
    "status": "pending",
    "group": "First Step"
  },
  "messages": [],
  "errors": []
}
```

#### task.list Response

```json
{
  "status": "ok",
  "operation": "task.list",
  "planPath": "docs/agent-context/current/implementation-plan.toml",
  "planRevision": "01K...",
  "tasks": [
    {
      "id": "01KX2JH7R1H4WJ6YJ2G8V0J3K9",
      "label": "Implement tests for Reactive Collections (RFC 0100)",
      "status": "pending",
      "group": "First Step"
    }
  ],
  "messages": [],
  "errors": []
}
```

#### task.complete Response

```json
{
  "status": "ok",
  "operation": "task.complete",
  "planPath": "docs/agent-context/current/implementation-plan.toml",
  "planRevision": "01K...",
  "task": {
    "id": "01KX2JH7R1H4WJ6YJ2G8V0J3K9",
    "label": "Implement tests for Reactive Collections (RFC 0100)",
    "status": "completed",
    "group": "First Step"
  },
  "messages": [],
  "errors": []
}
```

#### Error Response

```json
{
  "status": "error",
  "operation": "task.complete",
  "planPath": "docs/agent-context/current/implementation-plan.toml",
  "planRevision": null,
  "messages": [{ "level": "error", "text": "Task not found" }],
  "errors": [
    {
      "code": "TASK_NOT_FOUND",
      "message": "Task 'First Step::x' not found",
      "details": { "taskId": "First Step::x" }
    }
  ]
}
```

### Validation & Post-Conditions

- Tools MUST compute and return `planRevision` after writes.
- If `status != ok`, clients MAY call `task.list` and compare `planRevision` to confirm final state.
- If no write occurred, `planRevision` SHOULD be the current plan revision, not null.

## Amendment: Canonical Task IDs + Grouping

To eliminate ambiguity and output mismatch, task identifiers are **canonical and prefix-free**.

### Rules

- Each task stores a canonical ULID `id` generated internally.
- `group` is optional display-only grouping metadata (replaces `change.name` semantics).
- Task operations (`add`, `list`, `complete`, `remove`, `update`, `reorder`) accept and return canonical `id` only.
- `list` responses include both `id` and optional `group` for UI grouping without changing IDs.

### JSON Shape Additions

Task list entry and task response objects add `group` (optional):

```json
{
  "id": "01KX2JH7R1H4WJ6YJ2G8V0J3K9",
  "label": "Implement tests for Reactive Collections (RFC 0100)",
  "status": "pending",
  "group": "First Step"
}
```

### Migration Strategy

- When reading legacy tasks, map any `change.name` to `group`.
- When completing a task, accept legacy prefixed IDs and normalize to canonical ULID if unambiguous.
- Introduce a migration path to drop the `id` positional argument and allow `exo task add --label <label> [--group <group>]` only.

## Drawbacks

- Requires tool clients to adopt structured outputs and potentially adjust parsers.
- Introduces an additional hashing step for each tool call.

## Alternatives

- Continue to rely on human-readable output plus client-side parsing (fragile).
- Provide optional JSON output only (still ambiguous without strict schema).

## Unresolved Questions

- What hash canonicalization method should be used for TOML (strict re-serialize vs raw bytes)?
- Should `planRevision` be a stable hash or a ULID per write?

## Future Possibilities

- Extend the contract to phases, ideas, and inbox tools.
- Add a "verify" operation that returns a diff between expected and actual plan state.

# RFC 178: Instruction Localization Convention

| Field | Value |
|-------|-------|
| **Stage** | 1 (Proposal) |
| **Created** | 2025-01-30 |
| **Champion** | — |
| **Related** | RFC 0111 (Agent Guidance Architecture), RFC 0132 (CommandSpec Unification) |

## Summary

Establish a convention for writing instruction file content that:
1. Localizes guidance to the consumer's interface (LM tools for agents, CLI for humans)
2. Uses affirmative framing (what TO do) rather than prohibitive framing (what NOT to do)
3. Leads with concepts before introducing tools
4. Distinguishes content (editable) from metadata/state (tool-managed)

## Motivation

### The Abstraction Leakage Problem

Current instruction files leak implementation details that confuse agents:

```markdown
# ideas.toml - Do Not Edit

This file is **managed by Exosuit**. Direct edits bypass ULID generation and validation.
```

This framing has several problems:

1. **Negative framing creates confusion**: "Do Not Edit" tells agents what they *can't* do, but agents need to know what they *should* do. The prohibition implies the file exists as something editable that's being restricted.

2. **Implementation detail exposure**: "ULID generation and validation" are internal mechanisms. Agents don't need to know *why* direct edits are problematic—they need to know the right way to accomplish their goal.

3. **File-centric rather than concept-centric**: Leading with "This file is managed by..." frames the interaction around storage format rather than the concept being manipulated.

### The Projection Insight

RFC 0111 establishes that LM tools and CLI are both **projections** of the same CommandSpec:

```
┌─────────────────────────────────────────────┐
│           command-spec.json                 │
│   (Canonical definition of operations)      │
└─────────┬───────────────────┬───────────────┘
          │                   │
          ▼                   ▼
┌─────────────────┐   ┌─────────────────┐
│   LM Tools      │   │      CLI        │
│   (VS Code)     │   │   (Universal)   │
└─────────────────┘   └─────────────────┘
```

Instruction files are a **third projection**—guidance localized to the LM agent's context. Just as CLI help text references `exo phase start` and LM tool descriptions reference `exo-phase-start`, instruction files should reference the interface appropriate to their consumer.

### Current vs. Desired

| Aspect | Current | Desired |
|--------|---------|---------|
| Framing | "Do not edit this file" | "Ideas are managed through..." |
| Focus | File as storage | Concept being manipulated |
| Interface | Mixed (mentions "tools" generically) | Localized (specific LM tool names) |
| Tone | Prohibitive | Affirmative |

## Detailed Design

### Terminology

- **Localization**: Adapting guidance to reference the interface available to the consumer (LM tools for agents in VS Code, CLI for terminal users).
- **Affirmative framing**: Stating what to do rather than what not to do.
- **Concept-first**: Leading with the semantic meaning before introducing the mechanism.

### The Four Conventions

#### 1. Localization Principle

Instruction files consumed by LM agents must reference LM tools, not CLI commands.

**Before:**
```markdown
Use `exo idea add "My idea"` to add new ideas.
```

**After:**
```markdown
Use the `exo-idea` tool to capture new ideas.
```

The same guidance, localized for CLI users (e.g., in `--help` output), would say:
```
Use `exo idea add "My idea"` to add new ideas.
```

#### 2. Affirmative-Only Framing

Don't tell agents what they can't do. Tell them what they should do.

**Before:**
```markdown
# ideas.toml - Do Not Edit

This file is **managed by Exosuit**. Direct edits bypass ULID generation and validation.

**Use LM tools instead:** `exo-idea` to add ideas, `exo-status` to view.
```

**After:**
```markdown
# Ideas

Ideas capture rough thoughts, feature requests, and improvements for later triage.

**To work with ideas:**
- Add a new idea: `exo-idea` tool
- View current ideas: `exo-status` tool
```

The existence of `ideas.toml` as the storage mechanism is an implementation detail. The agent is working with *ideas*, not with a file.

#### 3. Concept-First Structure

Lead with what the concept IS, then provide the tools for working with it.

**Structure:**
```markdown
# [Concept Name]

[What this concept represents—one sentence.]

**To work with [concept]:**
- [Action verb]: `[exo-tool-name]` tool
- [Action verb]: `[exo-tool-name]` tool
```

**Example:**
```markdown
# Project Plan

The plan defines the project's epochs, phases, and tasks—the roadmap of work.

**To work with the plan:**
- Check current status: `exo-status` tool
- View phase details: `exo-phase` tool
- Start a phase: `exo-phase-start` tool
- Finish current phase: `exo-phase-finish` tool
- Add a task: `exo-add-task` tool
- Complete a task: `exo-task-complete` tool
```

#### 4. Content vs. Metadata Distinction

Some artifacts have both **content** (prose, code—directly editable) and **metadata/state** (managed by tools).

**RFC Example:**

```markdown
# RFCs

RFCs document design decisions and their rationale.

**RFC prose content** (Summary, Motivation, Design sections) can be edited directly—
it's content, like source code.

**RFC lifecycle** (stage promotion, metadata) is managed through tools:
- Create an RFC: `exo-rfc-create` tool
- Promote to next stage: `exo-rfc-promote` tool
- List RFCs: `exo-rfc-list` tool
```

This acknowledges that agents CAN and SHOULD edit RFC prose while using tools for state transitions.

### Anti-Patterns to Avoid

| Anti-Pattern | Example | Problem |
|--------------|---------|---------|
| Prohibition-first | "Do not edit this file" | Tells what NOT to do |
| Implementation leakage | "Bypasses ULID generation" | Exposes internal details |
| File-centric framing | "This file is managed by..." | Focuses on storage, not concept |
| Mixed interfaces | "Use `exo-idea` or `exo idea add`" | Creates decision paralysis |
| Passive voice | "Ideas are stored in this file" | Obscures the action |

### Steering Output Localization

The localization principle extends beyond instruction files to all agent-facing output:

| Channel | Consumer | References |
|---------|----------|------------|
| `.instructions.md` files | LM agents | LM tools (`exo-status`) |
| CLI `--help` output | Terminal users | CLI commands (`exo status`) |
| `exo-steering` tool output | LM agents | LM tools |
| Error messages (CLI) | Terminal users | CLI commands |
| Error messages (LM tool) | LM agents | LM tools |

### Implementation Details

#### Instruction File Template

```markdown
---
applyTo: "**/path/to/files"
---

# [Concept Name]

[What this concept represents—one sentence.]

**To work with [concept]:**
- [Action verb]: `[exo-tool-name]` tool
- [Action verb]: `[exo-tool-name]` tool

[Optional: Additional context about when/why to use these tools]
```

#### Validation Checklist

When reviewing instruction file content:

- [ ] Does it lead with the concept, not the file?
- [ ] Does it use affirmative framing (what to do)?
- [ ] Does it reference LM tools (not CLI commands)?
- [ ] Does it avoid implementation details (ULIDs, validation, file format)?
- [ ] Does it distinguish content from metadata where applicable?

## Drawbacks

1. **Rewrite effort**: Existing instruction files need updating.
2. **Subtle distinction**: The difference between "don't edit" and "use this tool" may seem minor, but the cognitive framing matters for agent behavior.
3. **Maintenance overhead**: Two phrasings (LM tool vs CLI) for the same operation.

## Alternatives

### Alternative 1: Keep Prohibition Framing

Continue with "Do not edit" style. This is explicit but:
- Creates adversarial framing
- Doesn't teach the right behavior
- Leaks implementation details

### Alternative 2: Unified Interface References

Reference operations abstractly without specifying tool or CLI:
```markdown
Use the "add idea" operation to capture new ideas.
```

Rejected because:
- Agents need concrete, actionable tool names
- Abstraction adds cognitive overhead
- Defeats the purpose of localization

## Unresolved Questions

1. **Automation**: Should we generate instruction files from CommandSpec to ensure consistency?
2. **Validation**: Should a linter enforce these conventions?
3. **Versioning**: How do we handle instruction content when tools are renamed?

## Future Possibilities

1. **Generated instructions**: CommandSpec could include instruction templates that are automatically localized for each projection (LM tool, CLI help, instruction files).

2. **Dynamic localization**: Instruction content could be generated at runtime based on detected context (VS Code vs. terminal vs. web).

3. **Instruction testing**: Automated tests that verify instruction files follow conventions and reference valid tool names.

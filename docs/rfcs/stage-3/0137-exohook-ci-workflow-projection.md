# RFC 137: Exohook CI Workflow Projection


# RFC 0137: Exohook CI Workflow Projection

## Summary

Extend exohook's projection system to generate GitHub Actions workflows from `hooks.toml`, ensuring CI configuration stays in sync with local validation lanes.

## Motivation

### The Problem

Exohook has a "projections" model where `hooks.toml` derives artifacts:

| Projection   | Status         | Output                                         |
| ------------ | -------------- | ---------------------------------------------- |
| Git hooks    | ✅ Implemented | `.git/hooks/pre-commit`, `.git/hooks/pre-push` |
| CI workflows | ❌ Missing     | `.github/workflows/*.yml`                      |

Currently, **there is no CI configured at all**. The repository has no `.github/workflows/` directory. This means:

- PRs have no status checks
- Main branch has no protection
- Tests only run if someone remembers to push (and has git hooks installed)
- The `ci` lane exists but only runs locally

### The Vision

From RFC 0081:

> `hooks.toml` is general-purpose: it must contain enough information to derive CI steps.

The same declarative configuration that drives git hooks should drive CI:

```toml
[projections.git_hooks]
pre_commit = "coherence"
pre_push = "gate"

# NEW: CI projection
[projections.github_actions]
ci = "ci"  # Maps workflow name to lane
```

### Benefits

1. **Single source of truth** - Check definitions live in one place
2. **Consistency guarantee** - CI runs exactly what pre-push runs
3. **Reduced drift** - Adding a check to `hooks.toml` automatically updates CI
4. **Onboarding** - New contributors see the same validations locally and in CI

## Detailed Design

### Terminology

- **projection**: A derived artifact from `hooks.toml` (git hooks, CI workflows)
- **lane**: A validation context (dev, coherence, gate, ci)
- **workflow**: A GitHub Actions workflow file (`.github/workflows/*.yml`)

### User Experience (UX)

#### Generating Workflows

```bash
# Generate CI workflow from hooks.toml
exohook ci emit github-actions

# Output location defaults to .github/workflows/exo-ci.yml
# Can override with --output:
exohook ci emit github-actions --output .github/workflows/validate.yml

# Preview without writing (--dry-run):
exohook ci emit github-actions --dry-run
```

#### Configuration

```toml
[projections.github_actions]
# Map workflow names to lanes
ci = "ci"

# Optional: additional workflows for different triggers
# pr = "coherence"  # Run coherence checks on PRs (future)
```

#### Generated Workflow Structure

The generator emits **one job per check** in the lane:

- Each check becomes its own GitHub Actions job
- Each job sets up only the toolchain it needs (inferred from the check's `run` command)
- Jobs run in parallel through GitHub's job concurrency model
- The workflow runs check commands directly; it does **not** build or invoke exohook in CI

#### Auto-Regeneration and Opt-Out

By default, workflow projection is **auto-regenerated** after any successful exohook command
except `ci emit` itself. The regeneration behavior is safe by construction:

- Only files with the `# AUTO-GENERATED by exohook` header are overwritten
- Files without that header are left untouched
- If the target file does not exist, it is created

To opt out, set:

```toml
[projections.github_actions]
enabled = false
```

Additionally, removing the auto-generated header from the workflow file prevents
future overwrites.

#### Generated Workflow

For the current `ci` lane:

```toml
[lane.ci]
label = "CI (HEAD)"
scope = { op = "base", base = "head" }
checks = ["test", "rust-coverage"]
```

Generates `.github/workflows/exo-ci.yml`:

```yaml
# AUTO-GENERATED by exohook
# Source: .config/exo/hooks.toml [lane.ci]
# To opt out: remove this header and set [projections.github_actions] enabled = false
# Regenerate with: exohook ci emit github-actions

name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  CARGO_TERM_COLOR: always

jobs:
  test:
    name: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 9
      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: "22"
          cache: "pnpm"
      - name: Install dependencies
        run: pnpm install
      - name: Run check
        run: pnpm -r run test:unit

  rust-coverage:
    name: rust-coverage
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Rust
        uses: dtolnay/rust-action@stable
        with:
          components: llvm-tools-preview
      - name: Cache cargo
        uses: Swatinem/rust-cache@v2
      - name: Install cargo-llvm-cov
        uses: taiki-e/install-action@cargo-llvm-cov
      - name: Run check
        run: cargo llvm-cov --workspace --lcov --output-path lcov.info
```

### Architecture

#### New CLI Subcommand

```
exohook ci emit <target>

TARGETS:
  github-actions    Generate GitHub Actions workflow

OPTIONS:
  --output <path>   Output file path (default: .github/workflows/exo-ci.yml)
  --dry-run         Print to stdout instead of writing file
  --lane <id>       Override lane (default: from projections.github_actions)
```

#### Implementation Flow

```
exohook ci emit github-actions
    ↓
Load .config/exo/hooks.toml
    ↓
Read [projections.github_actions] → find lane mapping
    ↓
Read [lane.ci] → get check list
    ↓
Read [check.*] for each check → get commands, dependencies
    ↓
Infer toolchain requirements (Rust, Node, pnpm, cargo-llvm-cov)
    ↓
Generate YAML from template
    ↓
Write to .github/workflows/exo-ci.yml
```

#### Toolchain Detection

The generator inspects check `run` commands to detect required tools:

| Pattern in `run` | Inferred Requirement  |
| ---------------- | --------------------- |
| `cargo`          | Rust toolchain        |
| `pnpm`           | pnpm + Node.js        |
| `npm`            | Node.js               |
| `cargo llvm-cov` | cargo-llvm-cov action |
| `cargo fmt`      | rustfmt component     |
| `cargo clippy`   | clippy component      |

This avoids requiring explicit toolchain metadata in `hooks.toml`.

#### Caching

Generated workflows include Cargo caching (e.g., a rust-cache action) when Rust toolchains
are inferred. This speeds up repeated CI runs without requiring extra user configuration.

### Phased Rollout

#### Phase 1: Bootstrap (Immediate)

Generate `.github/workflows/exo-ci.yml` with one job per check. This unblocks CI immediately
without requiring a separate hand-written workflow.

#### Phase 2: Generation Command

Implement `exohook ci emit github-actions` that generates the workflow from `hooks.toml`.

#### Phase 3: Sync Check

Add `exohook ci check` that verifies the generated workflow matches what would be generated from current `hooks.toml`. This can run in CI itself to detect drift.

#### Phase 4: Additional Targets (Future)

- GitLab CI (`.gitlab-ci.yml`)
- CircleCI (`.circleci/config.yml`)
- Generic shell script

## Drawbacks

1. **Generated files are awkward** - Some teams prefer hand-written CI configs
2. **Template rigidity** - Complex CI needs may not fit the template
3. **Toolchain detection heuristics** - May miss edge cases or detect incorrectly

Mitigation: The generated workflow runs `exohook validate ci`, which is the source of truth. The workflow is just the bootstrap to get there.

## Alternatives

### Alternative A: Pure Manual Workflow

Just write `.github/workflows/ci.yml` by hand and keep it in sync manually.

**Rejected because**: Defeats the purpose of declarative configuration. Drift between local hooks and CI is the core problem we're solving.

### Alternative B: Reusable Workflow Library

Publish a reusable GitHub Actions workflow that projects can reference:

```yaml
jobs:
  validate:
    uses: exosuit/workflows/.github/workflows/validate.yml@v1
    with:
      lane: ci
```

**Deferred**: Good idea but requires publishing infrastructure. Can layer on top of local generation later.

### Alternative C: GitHub App

A GitHub App that reads `hooks.toml` and runs checks directly, without generating workflow files.

**Rejected because**: Adds external dependency, doesn't work for other CI systems, harder to debug.

## Unresolved Questions

1. **Matrix builds?** Should there be support for testing on multiple OS/versions?
   - Defer to future RFC

2. **Self-update?** Should CI regenerate the workflow and fail if it differs?
   - Yes, via `exohook ci check` in Phase 3

## Future Possibilities

1. **PR-specific lane** - Run `coherence` on PRs (faster), `gate` on merge to main
2. **Artifact upload** - Upload `lcov.info` as build artifact
3. **Status checks per check** - Report each check as a separate GitHub status
4. **Branch protection rules** - Generate branch protection config from lanes
5. **Dependabot integration** - Auto-update action versions in generated workflow

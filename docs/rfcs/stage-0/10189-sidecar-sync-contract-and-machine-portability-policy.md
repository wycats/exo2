# RFC 10189: Sidecar Sync Contract and Machine Portability Policy

**Status:** Idea
**Feature:** sidecar

## Summary

Sidecars are the portable state boundary for Exosuit projects.

Today, the idea is right but the contract is still implicit. Exo can store
project state in a sidecar repository, but users and agents can still encounter
raw Git failures, stale runtime processes, dirty SQL projections, or sidebar
state that does not match the active MCP/daemon state.

This RFC defines the sidecar contract: Exo commands are the only normal
interface, SQLite is the local canonical state, the SQL projection is the
portable representation, and Exo is responsible for committing, pushing,
syncing, and explaining sidecar state in product terms.

It also separates two ideas that current dogfooding has shown must stay
separate: local durability and portability. A command may durably update the
canonical SQLite database before it returns, including by recording activity
events. That does not mean the same command should wait for SQL projection,
Git commit, fetch, merge, or push work before returning.

## Problem and Thesis

A sidecar-backed project should feel like one Exo project that follows the user
between machines.

At the moment, sidecar dogfooding shows that the user still has to know too
much about the implementation:

- which SQLite database the active runtime is using
- which daemon or MCP process is serving the workspace
- whether a machine-channel read wrote durable activity rows
- whether the SQL projection has been flushed
- whether the sidecar Git repository needs fetch, push, sync, or manual repair
- whether a ref-lock failure means real divergence or stale remote-tracking
  state

These are Exo's responsibilities. The user should not need to inspect the
sidecar repository with raw Git commands during ordinary work, and the agent
should not need to reason about sidecar internals to decide whether project
state is trustworthy.

The thesis of this RFC is:

> Sidecar Git is a transport for Exo state, not the state model itself.

The state model remains Exo's model: phases, goals, tasks, RFCs, inbox items,
activity, and runtime identity. Git is how durable state moves between
machines. When Git disagrees, Exo should translate that disagreement into Exo
state and Exo recovery actions.

Current dogfooding shows the gap concretely: sandboxd SQL projection rows were
written into the shared sidecar checkout and remained uncheckpointed, then exo2
later observed uncommitted sidecar checkout state while doing unrelated work.
That is evidence that the checkpoint boundary failed at the original write
path. It should drive implementation toward project-scoped checkpoint
guarantees, sidecar projection merge setup, and semantic sync reporting.

## State Model

A sidecar-backed project has three layers:

```text
Exo surfaces
  CLI / daemon / MCP / machine-channel / VS Code sidebar
        |
        | project resolve
        v
Canonical project database
  SQLite state used by the active Exo runtime
        |
        | checkpoint / import
        v
Sidecar repository
  Git-friendly SQL projection plus ignored local runtime state
```

The resolved SQLite database is canonical for local command execution. The SQL
projection is the portable representation that Exo commits and syncs. The
sidecar repository is not a human editing surface and should not become a
second source of truth.

Durable sidecar state includes project memory:

- epochs, phases, goals, tasks, task logs, and verifications
- inbox rows, ideas, and RFC metadata
- durable agent and machine-channel activity that affects collaboration,
  perception, accountability, or debugging

Local runtime state is different:

- daemon sockets and PID files
- transient runtime directories
- sidebar and activation caches
- local probes that describe one machine's current process state

If a row is useful for project history or collaboration, it belongs in the
durable sidecar state. If it only helps one local process operate, it belongs
outside the portable sidecar state.

## Construction Invariants

Sidecar-backed write and exec mutations produce canonical SQLite state and
reach a local checkpoint before reporting mutation success. The portable SQL
projection is generated by Exo from SQLite; it is never a parallel authoring
surface.

Checkpointing flushes durable rows into the sidecar SQL projection, ensures
runtime and cache paths are ignored, and records a local sidecar commit for the
affected project. For sidecar-backed write and exec mutations, this checkpoint
barrier is part of the write path: it either reaches that local committed state
or returns semantic Exo sync state describing the project and write path that
still need attention before mutation success is reported.

Read-shaped commands may still record durable local activity rows without
waiting for projection, Git commit, fetch, merge, or push. Those rows become
portable through a quiet checkpoint worker or an explicit portability barrier.

Remote sync preserves compatible changes to different logical rows
automatically. Incompatible edits to the same logical row become semantic Exo
conflicts that identify the table, entity, and conflicting fields.

Sidecar status keeps runtime freshness, checkpoint progress, and remote sync
state as separate dimensions. Uncommitted sidecar checkout state is diagnostic
evidence for the checkpoint/sync layer to classify, attribute to a project, and
repair or report through Exo state.

## Persistence Contract

Sidecar persistence has three boundaries.

The first boundary is local durability. Exo writes the canonical SQLite
database before reporting that a local state change succeeded. This includes
ordinary mutations such as `task complete` and read-shaped commands that record
durable activity, such as a machine-channel `status` request that appends an
`agent_events` row. If the event matters for collaboration, perception,
accountability, or debugging, it belongs in SQLite immediately.

The second boundary is checkpointing. A checkpoint makes local durable state
portable by flushing SQLite rows into the sidecar SQL projection, ensuring
runtime/cache paths are ignored, and committing projection changes to the
sidecar repository.

For sidecar-backed write and exec mutations, local success includes both local
durability and local checkpointing for the affected project. Exo should not
report mutation success while the affected projection is dirty and
uncheckpointed. If the writer lane cannot acquire, repair, or verify the local
checkpoint path, the mutation should fail before reporting success.

For pure reads and read-shaped activity writes, local success only requires the
local durability boundary. Those commands may leave pending checkpoint work for
the quiet checkpoint worker or an explicit portability barrier.

The third boundary is remote portability. Remote portability fetches the
configured remote, semantically reconciles compatible projection changes, and
pushes when the remote can be safely advanced.

These boundaries are intentionally not the same operation for every command.
Pure reads may synchronously write local durable activity rows, but they must
not synchronously flush the SQL projection, create sidecar commits, fetch,
merge, or push before returning. Read latency is part of the Exo product contract:
`status`, `task list`, `goal list`, `phase status`, `plan read`,
`project resolve`, help/discovery calls, sidebar roots, and MCP reads should
return after local durability.

When a sidecar-linked project has a configured remote, the default remote
policy should be `if_remote`: Exo should try to keep checkpointed sidecar state
portable, but only at checkpoint or barrier boundaries. `if_remote` is not
permission for ordinary reads to perform remote Git work before returning.

Portability barriers are the commands that may wait for checkpoint and remote
sync work. The explicit barriers are:

- `sidecar repo sync` and related sidecar maintenance commands
- `dogfood verify`
- phase finishing when the phase contract requires portable state
- safe-to-move-machine or similar portability checks

A barrier follows the full portability path:

1. write canonical SQLite state
2. flush durable rows to the SQL projection
3. ensure runtime/cache paths are ignored
4. commit projection changes
5. fetch and semantically reconcile remote projection state
6. push when the remote can be safely advanced

If local durable state has not yet been checkpointed or pushed, Exo should
report that as pending portability debt. Pending checkpoint or sync debt is not
a failed read, and it should not be hidden by making ordinary reads wait for
Git work.

Exo may run a quiet checkpoint worker for a sidecar-backed project. The worker
is scoped to one resolved project/sidecar root, coalesces repeated activity
writes into useful commits, and holds the sidecar runtime lock only while it is
flushing, committing, reconciling, or pushing. It may opportunistically make
state portable, but it must use backoff and must surface blocked sync as Exo
state instead of turning ordinary reads into retries or waits.

If an event is not supposed to be durable, it should not be written to the
durable sidecar database. "Durable but ignored" is not a stable category.
"Durable but not yet portable" is a stable category, and it should be visible
as sidecar sync state.

Raw command activity follows RFC 10183's retention policy. Retention cleanup is
a local SQLite operation first; checkpointing later makes the retained event
set, or its cleaned-up absence, portable. This RFC does not introduce a new
agent-event compaction model.

## Push and Sync Contract

`sidecar repo push` is an Exo maintenance operation, not a thin wrapper around
`git push`.

Before deciding whether to push, Exo should fetch the configured remote. After
fetching:

- if the remote already contains the local intended commit, Exo reports that
  sidecar state is synced
- if local state can safely advance the remote, Exo pushes it
- if the remote contains state not present locally, Exo reports remote updates
  and routes to `sidecar repo sync`

`sidecar repo sync` is the semantic reconciliation path for multi-machine work
and the normal explicit portability barrier. It flushes and commits pending
local durable state, fetches remote state, compares SQL projection rows by table
and stable logical row id, preserves compatible non-overlapping rows, and stops
on incompatible edits to the same logical row.

Conflicts should be reported as Exo conflicts. A useful conflict report names
the table, stable id, title or label when available, conflicting fields, and
the next Exo action. Raw Git rejection text and raw SQL conflict hunks can be
diagnostic detail, but they should not be the normal user-facing explanation.

VS Code should surface semantic sidecar conflicts in the Sidecar pane and, when
human judgment is required, queue an agent-visible repair or review request
rather than silently applying a repository mutation from the sidebar.

## Runtime Identity Contract

Sidecar portability is only meaningful if all Exo activation paths agree on the
active project state.

The direct CLI, daemon CLI, MCP `exo-run`, machine-channel, and VS Code sidebar
must resolve the same sidecar database for the same workspace. If they do not,
Exo is in runtime drift: a stale daemon, stale MCP server, stale socket, or
sidebar cache is reporting against the wrong state.

Runtime drift is a local process problem, not a sidecar Git problem. Exo should
repair scoped runtimes for the current workspace before reporting that sidecar
state is healthy.

## Status Contract

Sidecar status should answer three separate questions:

1. Is the active Exo runtime using the expected project database?
2. Is local durable state waiting for a sidecar checkpoint?
3. Is checkpointed sidecar state portable with respect to the sidecar
   repository and remote?

Those questions should not collapse into a single dirty bit.

A clean sidecar worktree with a stale daemon is not dirty sidecar state. A
clean worktree that is one commit ahead is not fully portable. A missing remote
is different from a configured remote that cannot be reached. A push command
that reports success while status still reports `ahead=1` is an inconsistency
that Exo should make diagnosable.

Likewise, a clean worktree with recent local SQLite activity may still have
pending checkpoint debt. That should be reported as Exo sync state, not paid
for by making unrelated reads run sidecar Git operations.

Normal MCP and agent-facing status output should include only actionable
sidecar sync state. Full portability diagnostics belong in explicit
`sidecar status`, `sidecar repo status`, `dogfood verify`, JSON/debug output,
or sidebar diagnostic panes.

## Design Principles

### Exo Commands Are Authoritative

Humans and agents should use Exo commands, not raw SQLite or raw Git, for
ordinary sidecar state work.

### Product Language First

User-facing surfaces should say "sidecar state is synced," "another machine has
updates," "review sidecar conflict," or "Exo refreshed the workspace runtime."
They should not lead with ref-locks, remote-tracking branches, SQL projection
rows, or hidden runtime details.

### Local Repair Before User Burden

If Exo can prove that a local runtime is stale, it should restart the scoped
runtime once before asking the user or agent to do anything.

### Fast Reads Before Portability

Ordinary read commands should never wait for sidecar Git. They may record local
activity in SQLite, and they may schedule or request a later checkpoint, but
they should not serialize behind sidecar commits, fetches, merges, or pushes.

### Semantic Merge Before Raw Merge

Sidecar SQL projection files exist to make durable state portable. When two
machines change different logical rows, Exo preserves both rows through the
automated projection merge path. When they change the same logical row
incompatibly, Exo stops and explains the logical conflict.

## Incremental Delivery

### Phase 1: Separate status dimensions

Make status and dogfood verification distinguish runtime identity from sidecar
repository portability.

### Phase 2: Separate local activity from portability

Ensure durable machine-channel and command-event writes are recorded in SQLite
without making pure reads wait for sidecar projection or Git work. Replace
synchronous read-triggered auto-persist with scoped checkpoint debt and a
quiet, coalescing checkpoint worker.

### Phase 3: Add explicit checkpoint barriers

Make `sidecar repo sync`, dogfood verification, phase finish, and
safe-to-move-machine checks flush pending local durable state, commit the SQL
projection, reconcile remote state, and push when safe.

### Phase 4: Hide raw push failures

Harden `sidecar repo push` so stale refs, already-synced remotes, and remote
updates are reported as Exo states.

### Phase 5: Make semantic sync the recovery path

Make `sidecar repo sync` preserve non-overlapping durable rows, preserve durable
agent events, stop on same-row conflicts, regenerate canonical state, and push
clean automated merges.

### Phase 6: Add multi-machine dogfood gates

Dogfood should prove that another machine can push compatible state, this
machine can sync it, all Exo activation paths agree on the database, and raw
Git plumbing does not leak into the normal workflow.

### Follow-up: Sidecar write ownership

RFC 10191 defines the single-writer safety contract for shared sidecar
checkouts. This RFC defines how sidecar state becomes portable; RFC 10191
defines which runtime is allowed to perform automatic checkpoint and push work
when multiple machines may have active Exo processes.

### Follow-up: Epoch-owned collaboration

RFC 10192 defines the future concurrent collaboration model where ownership is
scoped to an epoch or workstream. This RFC's semantic sync contract is a
prerequisite for that model, but it does not define multi-owner collaboration
itself.

## Expected Outcomes

- Moving machines does not require manual sidecar Git inspection.
- Sidebar, MCP, daemon, and CLI agree on the active sidecar database.
- Read-shaped commands that record durable activity keep that activity durable
  in SQLite without waiting for sidecar Git.
- Pending checkpoint or remote sync work is surfaced as Exo sidecar state, not
  hidden inside ordinary read latency.
- Push and sync failures are explained as Exo state and Exo recovery actions.
- Dogfood can prove the workspace is using the same binary, runtime, database,
  and sidecar state across activation paths.

## Non-Goals

- Replacing the sidecar Git repository with a hosted service.
- Making raw SQL or raw Git conflict resolution the normal path.
- Automatically discarding durable sidecar state.
- Defining every remote conflict policy in this RFC.
- Defining sidecar writer ownership or stale writer fencing; see RFC 10191.
- Defining epoch-owned concurrent collaboration; see RFC 10192.
- Adding an aggressive background sync loop without a separate policy.
- Treating every read command as a sidecar Git sync barrier.

## Open Questions

1. What exact state file, SQLite table, or runtime marker should represent
   pending checkpoint debt?
2. How should the quiet checkpoint worker be started, stopped, and observed
   across CLI, daemon, MCP, and VS Code activation paths?
3. What is the first safe implementation boundary for phase finish: warn on
   pending portability debt, wait for full sync, or require an explicit
   sidecar barrier?

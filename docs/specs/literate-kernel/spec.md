# Literate Kernel Protocol

**Version:** 2.3.0
**Status:** Living Standard

## 1. Scope

This specification defines the **Literate Kernel Protocol**, a mechanism for embedding structured, executable instructions within a natural language stream generated by a Large Language Model (LLM). It defines:

1.  The **Lexical Grammar** for identifying "Cells" within a text stream.
2.  The **Abstract Machine** for processing these cells and transforming them into User Interface (UI) elements.
3.  The **Execution Model** for the "Re-Act" (Reason + Act) feedback loop, utilizing the **Adjacency Rule**.
4.  The **Security Model** for capability negotiation and prompt injection mitigation.

## 2. Conventions

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

## 3. Lexical Grammar

The protocol is defined in terms of the **RTD Source Grammar**. The Literate Kernel Protocol extends the standard RTD grammar by defining specific semantics for **XML Blocks** and **Inline XML Tags**.

### 3.1 Base Grammar

The stream MUST be parsed according to the [RTD Source Grammar](../rtd/syntax.md).

### 3.2 Cell Identification

The RTD Parser identifies **XML Blocks** starting with `<exo-` or `<tool_code>`. These blocks are mapped to **Cells** in the Literate Kernel Abstract Machine.

- **Block Cells**: Corresponds to `XML_BLOCK` tokens in the RTD Parser where the tag name is one of the defined Block Cell types (e.g., `exo-tool`, `exo-tree`).
- **Inline Cells**: Corresponds to `INLINE_XML` tokens (or equivalent inline handling) for tags like `exo-cmd`, `exo-link`.

**Constraint**: The parser MUST handle **Tail Buffering** as defined in the [RTD Streaming Protocol](../rtd/streaming.md) to ensuring that split tags (e.g., `<exo-to...`) are not emitted as text.

## 5. Cell Definitions

### 5.1 Block Cells

Block cells represent distinct UI components. They MUST NOT be nested within other cells.

| Cell Tag         | Attributes                     | Content                 | Behavior                                                                                              |
| :--------------- | :----------------------------- | :---------------------- | :---------------------------------------------------------------------------------------------------- |
| `<exo-tool>`     | `name` (Required)              | JSON Object (Arguments) | Buffers content, parses JSON (or JSONC), executes tool. **Constraint**: Arguments MUST be in content. |
| `<exo-tree>`     | `root` (Optional, default `/`) | JSON Array              | Renders native File Explorer tree.                                                                    |
| `<exo-edit>`     | `path` (Required)              | Unified Diff            | Renders interactive Diff View (Pending Edit).                                                         |
| `<exo-progress>` | (None)                         | Text String             | Emits progress indicator with text.                                                                   |

### 5.2 Inline Cells

Inline cells MAY appear within the text stream.

| Cell Tag     | Attributes                              | Content      | Behavior                                            |
| :----------- | :-------------------------------------- | :----------- | :-------------------------------------------------- |
| `<exo-cmd>`  | `id` (Required), `args` (Optional JSON) | Button Label | Renders clickable button executing VS Code command. |
| `<exo-link>` | `path` (Required)                       | Link Text    | Renders button/link opening file at `path`.         |
| `<exo-ref>`  | `path` (Required)                       | (None)       | Adds file to "Used References" UI.                  |

## 6. The Feedback Loop (Adjacency Rule)

The Feedback Loop defines the multi-turn interaction model for tool execution, governed by the **Adjacency Rule**.

### 6.1 The Batch

A **Batch** is a sequence of one or more `<exo-tool>` cells separated _only_ by **Ignorable Content** (Whitespace or XML Comments).

### 6.2 The Synchronization Barrier

The machine operates in an **Accumulating** mode by default.

1.  **Accumulation**: When an `<exo-tool>` is parsed, it is added to the current Batch. Execution is deferred.
2.  **Trigger**: The **Synchronization Barrier** is triggered when the machine encounters any character that is NOT:
    - Part of an `<exo-tool>` cell.
    - Ignorable Content (Whitespace/Comment).
    - **Clarification**: Other cell types (e.g., `<exo-tree>`, `<exo-cmd>`, `<exo-link>`) are considered "Non-Tool Content" and WILL trigger the barrier. Only `<exo-tool>` cells participate in the batch.
3.  **Action**: Upon triggering the barrier:
    - **Interrupt**: The machine MUST pause consumption of the stream (or discard subsequent tokens).
    - **Execute**: All tools in the current Batch are executed **Concurrently**.
    - **Await**: The machine waits for all tools to complete.
    - **Feedback**: Results are fed back to the LLM.
      - **Binding Note**: The specific transport (e.g., `role="tool"`, `role="user"`, or `role="system"`) is defined by the **Binding Protocol** (e.g., [VCOM](../vcom/spec.md) Section 5). The Kernel Abstract Machine is agnostic to the wire format.
      - **Constraint**: The Kernel **MUST immediately invoke the LLM** to generate a continuation response. The Kernel MUST NOT wait for user input.

### 6.3 The Hallucination Brake

If the Trigger character is **Text** (e.g., the model starts writing "The file contains..."), the machine MUST interpret this as a hallucination attempt. The machine SHALL interrupt the generation immediately, preventing the model from fabricating the tool output.

### 6.4 Context Persistence

To maintain context across turns without cluttering the user interface, the Kernel MUST persist tool outputs and other intermediate state.

- **Mechanism 1: Native Metadata (Preferred)**

  - If the host environment supports attaching metadata to a turn (e.g., VS Code `ChatResult.metadata`), the Kernel SHOULD store the raw tool output there.
  - **Visibility**: Invisible to the user.
  - **Reconstruction**: The Kernel retrieves this metadata when building the context for the next turn.

- **Mechanism 2: Hidden Comments (Fallback)**

  - If native metadata is unavailable, the Kernel MAY inject tool outputs into the chat stream using hidden XML comments.
  - **Format**: `<!-- EXO_TOOL_OUTPUT ... -->`
  - **Risk**: May be sanitized or rendered by some User Agents.

- **System Kick**: When injecting tool outputs (via either mechanism), the Kernel SHOULD append a **System Directive** to the end of the message to force the model to re-evaluate the context.

  - **Abstract Transport**: The directive is logically a "System Message".
  - **Binding**:
    - If the API supports a dedicated System role, use it.
    - If the API supports named User messages (e.g., VS Code `LanguageModelChatMessage.User(..., "system")`), use `role="user"` with `name="system"`.
    - **Fallback**: Only if neither is available, append `[System Notification]: ...` to the standard User message body.
  - **Content**: `Tool execution completed. Please continue with the user's request.`

- **Renderer Constraint**: The Agent MUST ensure that tool outputs are retrievable for future turns, either via the output stream or the host's metadata storage.

### 6.5 The "Silence" Protocol (UI-Superseded Content)

To prevent "Double Vision" (where the model textually summarizes data the user is already viewing in a rich UI), the Kernel MUST implement a **Silence Directive** for specific tools.

1.  **Trigger**: When the executed tool is known to render a **Rich UI Artifact** (e.g., `listDirectory`, `editFile`, `openFile`).
2.  **Mechanism**: The **System Kick** MUST be modified to explicitly forbid summarization.
    - _Template_: `Tool execution completed. The user has viewed the result in a rich UI. Do NOT summarize the data. Proceed immediately to analysis or the next step.`
3.  **Rationale**: This forces the model to treat the tool output as "Shared Context" rather than "New Information" that needs to be reported.

## 7. Security & Capabilities

### 7.1 Deprecation of `<exo-code>`

The `<exo-code>` tag is **DEPRECATED** and removed from this specification to prevent prompt injection attacks via fake closing tags. Implementations MUST use standard Markdown Fenced Code Blocks (```) for code display.

### 7.2 Capability Negotiation

The Kernel MUST NOT execute a tool unless its capability (e.g., `fs.write`, `shell.exec`) has been explicitly negotiated in the session handshake.

### 7.3 Intrinsic Tools

The following tools are part of the Kernel's Standard Library and are ALWAYS available (no negotiation required):

- `sys.help`: Returns documentation for available tools.
  - **Usage**: `<exo-tool name="sys.help">{}</exo-tool>`

## 8. The Standard Library

The following tools are defined as the "Standard Library" for the Literate Kernel. Implementations SHOULD provide these tools, subject to capability negotiation.

### 8.1 File System

- **`listDirectory`**: Lists the contents of a directory.
  - **Arguments**: `{ "path": string, "recursive"?: boolean }`
  - **Returns**: JSON Array of file entries or a File Tree UI.
- **`readFile`**: Reads the contents of a file for the Agent's context.
  - **Arguments**: `{ "path": string }`
  - **Returns**: The file content as a string.
  - **Note**: This tool is for the _Agent_ to read. It does not open an editor.

### 8.2 Editor Interaction

- **`openFile`**: Opens a file in the user's editor.
  - **Arguments**: `{ "path": string }`
  - **Returns**: Confirmation string.
  - **Note**: This tool is for the _User_ to see. It does not return content to the Agent.
- **`editFile`**: Proposes an edit to a file.
  - **Arguments**: `{ "path": string, "diff": string }`
  - **Returns**: A "Pending Edit" status. The edit is NOT applied immediately.

## Annex C: Compatibility Parsing (Normative)

To ensure robust operation with models trained on diverse datasets (as defined in [VCOM](../vcom/spec.md) Annex A), the Abstract Machine MUST implement the following "Lenient Parsing" rules. These rules take precedence over the strict Lexical Grammar in Section 3 when processing the input stream.

### C.1 Legacy Tags

The parser MUST recognize the following tags as valid **Block Cells**, even though they lack the `<exo-` prefix.

| Tag Name      | Behavior                                  | Emission                                 |
| :------------ | :---------------------------------------- | :--------------------------------------- |
| `thought`     | Collapsed reasoning block (CoT).          | `EmitCell("thought", {}, content)`       |
| `plan`        | Structured plan.                          | `EmitCell("plan", {}, content)`          |
| `antArtifact` | Anthropic-style artifact (Code/Markdown). | `EmitCell("antArtifact", attr, content)` |

**Parsing Rule**: These tags MUST be recognized only when they appear at the **start of a line**.

### C.2 Conflict Markers (Aider-style)

The parser MUST recognize "Search/Replace" blocks as valid **Block Cells**.

- **Start Pattern**: `<<<<<<< SEARCH` (at start of line)
- **End Pattern**: `>>>>>>> REPLACE` (at start of line)
- **Emission**: `EmitCell("edit", { format: "search_replace" }, content)`
- **Content**: The content includes everything between the start and end markers (inclusive or exclusive, depending on the VCOM binding expectation. Recommendation: Exclusive of markers, but preserving the internal `=======` separator).

### C.3 Hallucinated Tools

If the parser encounters a `<tool_code>` tag (a common legacy alias), it MUST treat it exactly as `<exo-tool>`.

## Annex A: Design Rationale (Informative)

1.  **Adjacency Heuristic**: We reject "Serial by Default" in favor of a model that respects the LLM's natural output structure. Tools grouped together are a batch; tools separated by text are a dependency chain.
2.  **JSON-as-Content**: Moving arguments to the content body allows for multi-line, validated JSON, solving the "Quoting Hell" of attribute-based arguments.
3.  **Hallucination Brake**: The "Interrupt-on-Text" mechanism serves as a security feature, preventing the model from assuming the output of a tool it hasn't yet run.
4.  **IDE Integration (v2.2.0)**: The distinction between `read_file` (Context) and `open_file` (UI) ensures the model acts as an "IDE Operator" rather than a text generator. The "Pending Edit" protocol (`edit_file` returning "Proposed") prevents accidental code destruction.

## Annex B: The System Prompt (Normative)

The following XML block defines the System Prompt that MUST be injected into the LLM context to enforce the Literate Kernel Protocol.

````xml
<system_instructions>
  <meta>
    <role>Literate Kernel</role>
    <version>2.2.0</version>
    <environment>VS Code (Integrated)</environment>
    <objective>Collaborative Technical Problem Solving via IDE Control</objective>
  </meta>

  <philosophy>
    1. **The IDE Operator**: You are not just a chat bot; you are a Remote Control for the IDE. You manipulate the Editor, Terminal, and File System to help the user.
    2. **Collaborative Reasoning**: When planning or explaining, be thorough. If requirements are ambiguous, ASK questions.
    3. **Efficient Execution**: When using tools, be terse. Optimize for token efficiency.
    4. **Reality**: You cannot "act" by describing an action. You must emit the `<exo-tool>` tag.
    5. **Closing the Loop**: Tool execution is an *intermediate* step. After a tool runs, the System will return control to you. You MUST immediately use the tool output to answer the User's original question. Do not stop until the User's goal is satisfied.
  </philosophy>

  <constraints>
    1. **NO SIMULATION**: Do not simulate tool outputs. The System will interrupt you immediately after you close a tool tag.
    2. **NO HALLUCINATION**: If you are unsure of a tool, use `sys.help`.
    3. **ADJACENCY RULE**:
       - Tools separated ONLY by whitespace/comments are executed in **PARALLEL**.
       - Tools separated by text are executed in **SERIES**.
    4. **COMPATIBILITY**: The `<tool_code>` tag is supported as a legacy alias for `<exo-tool>`.
    5. **NO ECHO**: If a tool renders a UI (like a file tree or diff), DO NOT describe it in text. Assume the user sees it.
  </constraints>

  <syntax_rules>
    1. **Tags**: Use `<exo-tool name="...">...</exo-tool>`.
    2. **Arguments**: Must be valid JSON (or JSONC) inside the tag content.
    3. **Code**: Use standard Markdown Fenced Code Blocks (```).
  </syntax_rules>

  <tool_guidelines>
    1. **Reading vs. Opening**:
       - Use `readFile` when YOU need to read code to understand it.
       - Use `openFile` when the USER asks to see code.
    2. **Editing**:
       - `editFile` proposes a change. It does not apply immediately.
       - Do not assume success until you receive the "User Accepted" signal.
  </tool_guidelines>

  <available_tools>
    {{TOOL_DEFINITIONS}}
  </available_tools>

  <examples>
    <example description="Parallel Batching">
      <thought>I need to analyze the frontend and backend structure.</thought>
      <content>
        <exo-tool name="listDirectory">
          {"path": "frontend"}
        </exo-tool>
        <exo-tool name="listDirectory">
          {"path": "backend"}
        </exo-tool>
      </content>
    </example>

    <example description="IDE Interaction (Open vs Read)">
      <user>Show me the main config file.</user>
      <assistant>
        I will open the config file for you.
        <exo-tool name="openFile">
          {"path": "src/config.ts"}
        </exo-tool>
      </assistant>
    </example>

    <example description="Edit Workflow (Pending State)">
      <user>Fix the typo in the header.</user>
      <assistant>
        I will propose a fix for the typo.
        <exo-tool name="editFile">
          {
            "path": "src/header.ts",
            "diff": "..."
          }
        </exo-tool>
        <!-- System Interrupts. User Reviews Diff. User Accepts. -->
        Edit accepted. The file has been updated.
      </assistant>
    </example>
  </examples>
</system_instructions>
````
